# AGENTS.md

## 1. 작업 시작 규칙

1. 항상 이 문서 규칙을 우선 적용한다.
2. 작업 시작 시 우선 확인 문서는 `TASKS.md`다.
3. 아래 문서는 작업 상황에 따라 확인한다.
   - 새 세션 시작, 구조 파악 필요, 최근 구조 변경 가능성 있음: `project_context.md`
   - 현재 Phase 확인 또는 Phase 전환 판단 필요: `ROADMAP.md`
   - 최근 변경 맥락 확인 필요: `CHANGELOG.md`
4. `TASKS.md`의 현재 Phase와 다음 작업 범위 안에서만 구현한다.
5. `TASKS.md`에 없는 새로운 기능을 임의로 추가하지 않는다.
6. 현재 해야 할 작업에 집중한다.
7. 한 번에 너무 많은 기능을 추가하지 않는다.
8. 큰 기능은 `TASKS.md` 기준으로 여러 단계로 나누어 구현한다.
9. 실제 코드 수정 전에는 짧은 구현 계획을 먼저 설명한다.
   - 반복 소작업: 2~4줄 요약 또는 요청이 충분히 명확하면 바로 수정
   - 큰 기능 작업: 짧은 구현 계획 제시
10. `TASKS.md`는 항상 `ROADMAP.md`의 현재 Phase와 일치하게 유지한다.
11. 큰 기능 변화가 있으면 필요한 경우에만 `README.md`, `project_context.md`를 갱신한다.

## 2. 반복 소작업 속도 규칙

반복 소작업은 아래 범위를 기본으로 한다.
- 파일 1~3개 수정
- 작은 UI/문구 수정
- 버그 1개 수정
- 함수 1개 보완
- 테스트 1세트 추가 또는 수정
- 문서 1회 최소 갱신

반복 소작업일 때는 아래 규칙을 우선 적용한다.
- 프로젝트 전체를 재분석하지 않는다.
- `TASKS.md`와 현재 작업 관련 파일을 먼저 본다.
- 전체 디렉토리 스캔은 하지 않고 관련 파일만 확인한다.
- 현재 작업과 무관한 개선 아이디어는 제안만 하고 바로 구현하지 않는다.
- 과도한 리팩토링, 광범위한 정리, 파일 대량 이동, 네이밍 일괄 변경은 하지 않는다.
- 테스트는 전체 테스트보다 영향 범위 테스트를 우선한다.

예외 규칙
- 새 세션이거나 최근 구조 변경 가능성이 있는 경우에는 `project_context.md`와 최근 `CHANGELOG.md`를 우선 확인한다.
- 현재 작업의 맥락이 불명확하면 빠른 진행보다 문맥 확인을 우선한다.
- 공통 유틸, 상태관리, 설정 로직, 데이터 스키마, 인증/입출력 경계 코드를 수정하는 경우에는 소작업처럼 보여도 영향 범위를 넓게 판단한다.

## 3. 코딩 원칙

- 실제 실행 가능한 구조를 우선한다.
- 유지보수 가능하고 확장 가능한 파일 분리를 유지한다.
- 기능별 파일 분리를 기본 원칙으로 한다.
- 최소한의 의존성만 추가한다.
- 새 의존성이 필요하면 이유를 먼저 짧게 설명한다.
- 브라우저 계산 중심 구조를 유지한다.
- GAS는 인증, 원본 제공, 기록 저장 범위 안에서만 다룬다.
- 입력 오류, 데이터 누락, 예외 처리, 로그 기록을 고려한다.
- 모바일 우선 UI를 기본으로 한다.
- 가능한 경우 테스트 방법과 실행 명령어를 함께 남긴다.
- 기존 코드 스타일이 있으면 그 스타일을 우선 유지한다.

## 4. 파일 수정 규칙

- 기존 기능을 함부로 깨지 않는다.
- 큰 수정 전에는 현재 구조와 영향 범위를 먼저 짧게 판단한다.
- 큰 구조 변경 또는 다수 파일 수정이 필요한 경우 먼저 영향 범위를 설명한다.
- 기존 파일 수정 시 변경 범위를 최소화한다.
- 새로운 기능은 새 파일로 분리하는 것을 우선 고려한다.
- 비슷한 역할의 파일이 이미 있으면 그 위치를 우선 사용한다.
- 새 파일 생성이 필요할 때만 관련 폴더 구조를 확인한다.

## 5. 오류 대응 및 재분석 규칙

오류가 발생하면 아래 순서를 따른다.
1. 오류 메시지를 먼저 분석한다.
2. 최소 수정으로 해결 가능한지 판단한다.
3. 필요한 경우 로그를 추가한다.
4. 문제 해결 후 관련 테스트를 다시 실행한다.
5. 동일 오류가 반복되면 다른 접근 방식을 시도한다.
6. 필요한 경우에만 부분 재설계를 한다.
7. 기존 기능을 크게 변경하는 방식의 수정은 피한다.

재분석 규칙
- 오류가 발생해도 프로젝트 전체를 다시 읽지 않는다.
- 오류가 발생한 파일과 직접 관련된 코드 범위만 우선 분석한다.
- 필요한 경우에만 참조 파일을 추가 확인한다.

## 6. 작업 완료 보고 규칙

작업 완료 후 보고 형식은 작업 규모에 따라 나눈다.

### A. 반복 소작업 보고
1. 이번 작업 요약
2. 수정/생성 파일 목록
3. 사용자가 바로 확인할 실행 또는 테스트 포인트
4. `TASKS.md` 기준 다음 작업 제안

### B. 큰 기능 작업 또는 구조 변경 보고
1. 이번 작업에서 구현된 기능 요약
2. 생성되거나 수정된 파일 목록
3. 사용자가 검토해야 할 코드/설계 포인트
4. 사용자가 직접 실행해야 할 부분
   - 가능한 경우 실행 명령어 포함
5. 정상 동작 시 기대 결과
6. 현재 프로젝트 진행률 추정 퍼센트
7. `TASKS.md` 기준 다음 작업 제안

## 7. 수정 금지 규칙

아래 항목은 사용자가 직접 관리하므로 Codex는 수정하지 않는다.
- `skills_guide.xlsx`
- 외부 데이터 파일
- 사용자 설정 파일
- 실제 Google Sheets 데이터 내용

## 8. 개발 단계 관리 규칙

- 프로젝트에는 `ROADMAP.md`가 존재한다.
- 항상 현재 프로젝트가 어느 Phase인지 인식한다.

작업 순서
1. 필요 시 `ROADMAP.md`에서 현재 Phase를 확인한다.
2. `TASKS.md`는 현재 Phase 기준 작업만 포함하도록 유지한다.
3. 현재 Phase 작업이 완료되면
   - `ROADMAP.md`에서 다음 Phase로 이동한다.
   - `TASKS.md`를 다음 Phase 기준으로 갱신한다.
   - `CHANGELOG.md`에 Phase 전환 기록을 남긴다.

중요
- MVP Phase가 끝나도 프로젝트는 종료되지 않는다.
- 단계적으로 확장하여 최종 프로그램 완성까지 이어간다.
- `CHANGELOG.md`에는 아래도 기록한다.
  - 작업 날짜
  - 구현/수정 내용 요약
  - 테스트 결과 요약
  - Phase 변경 여부
  - 다음 작업 연결 메모

## 9. 중단 / 보류 / 보고 규칙

아래 경우 무리하게 계속 구현하지 말고 현재 상태를 정리해 보고한다.
- 필수 입력 정보가 부족한 경우
- 기존 구조와 충돌이 큰 경우
- 외부 연동 방식이 확정되지 않은 경우
- 테스트 실패 원인이 불명확한 경우
- 한 번의 작업 범위를 넘는 대규모 수정이 필요한 경우

보고 형식
1. 현재 막힌 원인
2. 이미 확인한 범위
3. 임시 해결 가능 여부
4. 권장 다음 작업
5. `TASKS.md` 반영 내용

## 10. 문서 동기화 규칙

- 코드가 바뀌면 필요한 문서만 최소 범위로 갱신한다.
- `TASKS.md`는 작업 상태가 바뀌었으면 반드시 갱신한다.
- 반복 소작업에서도 완료/진행중/다음 작업 변화가 생기면 `TASKS.md`를 최소 범위로 바로 반영한다.
- `ROADMAP.md`는 Phase 변경이 있을 때만 갱신한다.
- `README.md`는 설치/실행/사용 방법이 바뀐 경우 갱신한다.
- `project_context.md`는 구조/범위/입출력 정의가 바뀐 경우 갱신한다.
- `CHANGELOG.md`는 기능 추가, 버그 수정, 테스트 기준 변경, 작업 상태 변경, Phase 관련 변경 시 기록한다.
- 아주 작은 문구 수정, 주석 수정, 오타 수정은 `CHANGELOG.md` 기록을 생략할 수 있다.
- 단, 코드 동작에 영향을 주는 수정은 규모가 작아도 `CHANGELOG.md`에 1줄 이상 기록하는 것을 원칙으로 한다.

## 11. 이번 프로젝트 전용 규칙

- 최고 결과 기준 기본 스택은 다음과 같다.
  - React + Vite + TypeScript
  - Zustand
  - Tailwind CSS
  - Framer Motion
  - Vitest + React Testing Library
  - GAS Web App
  - Google Sheets
- 사용자 시트는 `lang_user_sheet` 별도 파일을 사용한다.
- 첫 언어는 일본어다.
- 다국어 확장 구조를 유지한다.
- 현재는 Phase 1이므로 다음 작업 범위를 넘는 고급 기능은 미룬다.
- 점수/콤보/하트/복습 계산은 프론트에서 처리한다.
- 기록 저장은 세션 종료 시 batch 방식으로 처리한다.
